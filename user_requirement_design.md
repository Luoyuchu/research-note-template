# User Requirement Design

这个文件是根据 `docs/archive/prototype_conv_logs.txt` 中的原始对话提炼出来的需求摘要。

目标不是复述全部对话，而是把真正稳定、可执行、值得长期维护的要求收敛成一套仓库设计原则。

## 一句话目标

构建一个低维护、低阻力、适合科研记录的单仓库系统：

- **本地写作** 用 Obsidian
- **版本同步** 用 GitHub
- **发布渲染** 用 Quartz + Vercel
- **图片托管** 用 PicList + 对象存储
- **访问控制** 可选 StatiCrypt

## 原始需求中的核心矛盾

### 1. 写作必须低阻力

用户不希望每天把信息拆碎、分类、搬运后才开始写。

因此模板必须鼓励：

- 先写 Daily Log
- 再沉淀为项目页、知识页、写作页

### 2. 单库优于多库

原始对话中多次出现“每个项目一个 vault 会切断知识连接”的担忧。

因此模板采用：

- 单个 vault
- 通过命名前缀和页面角色区分内容
- 通过 Obsidian / Quartz 的双向链接完成跨项目聚合

### 3. 图片不能进入 Git，但网站又必须能显示图片

这直接导出了一条强约束：

- 本地写作时保留本地图片链接
- 本地先上传到图床
- 发布构建时再把本地路径映射成最终发布 URL

### 4. AI 原始对话不应该污染核心知识网络

原始需求明确区分了：

- 值得提炼的正式知识
- 尚未消化但不想丢弃的 AI 原始材料

所以模板提供：

- `04_Resources/AI_Raw/` 作为隔离区
- 只在正式笔记中摘录结论并链接回原始材料

### 5. 项目过程和最终写作不能混在一起

研究工作最终常常会进入：

- advisor report
- paper draft
- proposal

因此模板需要单独保留一层 `Writing`：

- `Project` 负责执行过程
- `Writing` 负责交付表达
- 两者通过链接衔接，而不是互相复制结构

## 仓库级设计决策

### 1. 仓库中的 `vault/` 子目录就是写作根目录

这样做的理由：

- Obsidian 只需要面对真正的写作内容
- Git 与写作源文件仍然保持在同一仓库中
- `vault/.asset-registry/` 跟随同一仓库流转

### 2. 站点构建层放在隐藏目录 `.site/`

这样做的理由：

- Quartz 需要独立的 `content/` 构建目录
- 但构建层不应该污染写作视图
- 把它放进隐藏目录能减轻 file explorer 噪音

### 3. 发布前先做同步，而不是直接拿 `vault/` 给 Quartz

同步脚本承担两项职责：

1. 只挑出可发布内容
2. 把本地图片链接替换成 registry 中解析出的最终发布 URL

这能保证：

- 原始 vault 保持本地写作友好
- 发布内容保持干净和可控

### 4. 图片发布映射依赖 `vault/.asset-registry/`

不能靠猜。

尤其在 PicList 可能把 `png` 转成 `webp` 的情况下，只有 registry 里记录的上传结果路径，才是发布时可信的最终依据。

## 非目标

这个模板当前不追求：

- 多人实时协作
- 复杂权限系统
- CMS 式后台
- 自动把所有 AI 材料转成知识卡片

它优先服务的是：

- 单人或小团队
- 本地优先写作
- 静态发布
- 长期维护成本可控
